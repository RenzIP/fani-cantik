1. Pendahuluan
1.1 Tujuan Penulisan Dokumen
Dokumen Deskripsi Pengembangan Perangkat Lunak (DPPL) ini disusun sebagai panduan teknis komprehensif untuk pengembangan Sistem Informasi Dapur dan Restock Warung Nasi Bakar. Dokumen ini mendeskripsikan seluruh aspek perancangan dan pengembangan perangkat lunak, mulai dari deskripsi umum sistem, kebutuhan fungsional dan non-fungsional, desain arsitektur, perancangan data, hingga rencana pengujian dan spesifikasi produk.
Dokumen ini ditujukan untuk digunakan oleh:
• Pengembang perangkat lunak sebagai acuan teknis implementasi sistem.
• Manajer proyek untuk memantau cakupan dan kemajuan pengembangan.
• Penguji sistem untuk merancang dan melaksanakan skenario pengujian.
• Dosen pengampu sebagai evaluasi tugas besar mata kuliah Rekayasa Perangkat Lunak.
• Pemilik Warung Nasi Bakar sebagai pengguna akhir dan pemangku kepentingan.

1.2 Lingkup Masalah
Perangkat lunak yang dikembangkan adalah Sistem Informasi Dapur dan Restock Warung Nasi Bakar. Sistem ini merupakan aplikasi berbasis web yang dirancang khusus untuk menggantikan seluruh proses manual pengelolaan dapur dan stok bahan baku pada Warung Nasi Bakar dengan alur digital yang terstruktur.
Sistem ini akan:
• Mendigitalisasi pencatatan stok bahan baku, pengecekan kedaluwarsa, dan proses restock secara terintegrasi.
• Mengintegrasikan alur pesanan dari kasir ke dapur melalui antrian pesanan digital secara real-time.
• Menyediakan laporan stok, produksi, dan restock yang dapat diakses pemilik melalui dashboard.
• Mengurangi risiko operasional akibat proses manual dan meningkatkan efisiensi kerja dapur.
Sistem ini tidak mencakup integrasi dengan platform pemesanan online, fitur pemesanan mandiri pelanggan (self-ordering), fitur akuntansi lanjutan, integrasi e-faktur, manajemen supplier otomatis, maupun aplikasi mobile.

2. Deskripsi Umum Perangkat Lunak
2.1 Deskripsi Umum Desain
Sistem Informasi Dapur dan Restock Warung Nasi Bakar dirancang dengan pendekatan desain terstruktur berbasis aliran data (Data Flow-based Design). Pendekatan ini dipilih karena sistem ini bersifat transaction-processing dengan aliran data yang jelas dan terdefinisi antara kasir, dapur, dan admin/pemilik.
Fitur-fitur penting yang menjadi inti perancangan meliputi:
• Modul Kelola Bahan Baku: Pencatatan dan monitoring stok bahan baku dengan status otomatis (Aman/Hampir Habis/Kedaluwarsa).
• Modul Notifikasi Stok: Peringatan otomatis jika stok di bawah minimum atau bahan mendekati kedaluwarsa.
• Modul Kelola Pesanan: Form input pesanan digital oleh kasir dengan antrian real-time ke dapur.
• Modul Kelola Produksi: Pengelolaan status pesanan oleh dapur dan pengurangan stok otomatis setelah produksi selesai.
• Modul Kelola Restock: Pencatatan penambahan stok bahan baku oleh dapur.
• Modul Laporan: Dashboard admin dengan laporan stok, produksi harian, dan restock.
• Modul Keamanan: Autentikasi berbasis role (Kasir / Dapur / Admin) dengan hak akses berbeda.

2.2.1 Antarmuka Sistem
Sistem Informasi Dapur dan Restock berinteraksi dengan dua antarmuka sistem utama:
• Antarmuka Layar Dapur: Sistem mengirimkan antrian pesanan ke perangkat/layar dapur melalui koneksi LAN lokal (WebSocket atau HTTP polling) secara real-time.
• Antarmuka Notifikasi: Sistem mengirimkan peringatan stok minimum dan kedaluwarsa ke halaman dashboard admin dan dapur secara otomatis.

2.2.2 Antarmuka Pengguna
Antarmuka pengguna dirancang sederhana dan intuitif. Karakteristik antarmuka:
• Navigasi berbasis peran: tampilan dashboard berbeda untuk kasir, dapur, dan admin.
• Form pesanan menampilkan pilihan menu, topping, dan level pedas dalam format yang mudah diisi.
• Halaman antrian dapur menampilkan daftar pesanan secara real-time dengan tombol aksi status (Proses/Selesai).
• Dashboard admin menyajikan ringkasan stok, notifikasi, dan akses cepat ke laporan.
• Pesan validasi dan peringatan tampil dalam bahasa Indonesia dengan instruksi yang jelas.

2.2.3 Antarmuka Hardware
Perangkat keras yang dibutuhkan dan diantarmukakan oleh sistem:
• Perangkat Kasir: PC/Tablet dengan RAM minimal 2GB, browser modern (Chrome, Firefox, Edge versi terbaru).
• Layar Dapur: Perangkat display (tablet/monitor) yang terhubung ke jaringan LAN lokal yang sama dengan server.
• Perangkat Admin: PC/Laptop dengan browser modern untuk mengakses dashboard dan laporan.

• Kasir: Login ke sistem, membuka form pesanan, mengisi data pesanan pelanggan, dan mengirimkan ke antrian dapur.

• Admin/Pemilik: Login ke dashboard, mengelola data bahan baku, memantau notifikasi stok, dan mengakses laporan operasional.
• Laporan stok dapat diakses kapan saja; sistem memantau stok dan kedaluwarsa secara otomatis selama aplikasi berjalan.

2.5 Batasan
• Sistem harus berjalan pada perangkat dengan spesifikasi menengah ke bawah (RAM minimal 2GB, browser modern) tanpa instalasi khusus yang kompleks.
• Anggaran pengembangan disesuaikan dengan skala UMKM; tidak menggunakan infrastruktur cloud premium.
• Pengembangan harus selesai dalam satu semester akademik.
• Pada rilis pertama, sistem berdiri sendiri (standalone) tanpa integrasi ke platform pemesanan online eksternal.
• Sistem harus mampu menangani minimal 150 pesanan dan 50 transaksi restock per hari tanpa penurunan performa.
• Sistem harus tersedia minimal 99% selama jam operasional warung.

3. Deskripsi Umum Kebutuhan
3.1 Kebutuhan Antarmuka Eksternal
3.1.1 Antarmuka Pemakai
User interface untuk mengoperasikan Sistem Informasi Dapur dan Restock menggunakan perangkat input standar:
• Keyboard dan mouse untuk PC/laptop kasir dan admin.
• Layar sentuh (touchscreen) untuk tablet kasir dan layar dapur.
• Antarmuka berbasis browser web yang responsif dan intuitif.

3.1.2 Antarmuka Perangkat Keras
• Network Interface Card (NIC): Diperlukan pada setiap perangkat untuk koneksi jaringan LAN lokal.
• Layar Dapur: Tablet atau monitor yang terhubung ke LAN untuk menampilkan antrian pesanan.

3.1.3 Antarmuka Perangkat Lunak
• Web Browser API: Untuk rendering antarmuka pengguna berbasis HTML5/CSS3/JavaScript.
• REST API: Antarmuka komunikasi antara frontend (browser) dan backend server.
• WebSocket API: Untuk notifikasi real-time antrian pesanan ke layar dapur.

3.1.4 Antarmuka Komunikasi
• HTTP/HTTPS (port 80/443): Untuk komunikasi request-response antara browser klien dan web server.
• WebSocket (ws://): Untuk komunikasi real-time notifikasi pesanan ke dapur.
• TCP/IP: Protokol jaringan dasar pada LAN lokal.

3.2 Deskripsi Fungsional
3.2.1 Context Diagram
Context Diagram Sistem Informasi Dapur dan Restock Warung Nasi Bakar menggambarkan hubungan antara sistem dengan empat entitas eksternal utama:
• Kasir: Mengirimkan data pesanan pelanggan ke sistem; menerima konfirmasi pesanan diterima dan status antrian.
• Dapur: Menerima notifikasi antrian pesanan dari sistem; mengirimkan data status produksi dan restock bahan.
• Admin/Pemilik: Mengirimkan data bahan baku dan permintaan laporan; menerima laporan stok, produksi, restock, dan notifikasi peringatan.
• Supplier: Pihak eksternal yang memasok bahan baku; tidak memiliki akses langsung ke sistem.
Proses utama dalam Context Diagram adalah SISTEM INFORMASI DAPUR DAN RESTOCK yang mengelola seluruh aliran data antara keempat entitas tersebut.

4.2.2 Conceptual Data Model
Conceptual Data Model (CDM) menggambarkan entitas dan relasinya secara konseptual tanpa mempertimbangkan implementasi fisik database. Entitas utama beserta relasinya:
• PENGGUNA berelasi MEMBUAT dengan PESANAN (satu pengguna kasir dapat membuat banyak pesanan).
• PESANAN berelasi MENGHASILKAN dengan PRODUKSI (satu pesanan menghasilkan satu record produksi).
• BAHAN_BAKU berelasi MEMILIKI dengan RESTOCK (satu bahan baku dapat memiliki banyak riwayat restock).
• PRODUKSI berelasi MENGGUNAKAN dengan BAHAN_BAKU (banyak produksi menggunakan banyak bahan baku - relasi N:M).

4.2.3 Physical Data Model
Physical Data Model (PDM) merupakan penjabaran CDM ke dalam struktur tabel database dengan tipe data, primary key, foreign key, dan constraint yang spesifik. PDM ini digunakan sebagai acuan pembuatan script DDL database.
Tabel-tabel dalam PDM: t_pengguna, t_bahan_baku, t_pesanan, t_produksi, t_restock, t_produksi_bahan.

5.2 Deskripsi Fungsional secara Rinci
5.2.1 Spesifikasi Fungsi: Kelola Bahan Baku (FR-001 s.d. FR-004)
Identifikasi/NamaF-001: Kelola Bahan BakuDeskripsi IsiAdmin memasukkan data bahan baku; sistem memantau stok dan memberikan peringatan otomatis jika stok di bawah minimum atau bahan mendekati/melewati kedaluwarsa.JenisForm Entry + Proses Otomatis (Notifikasi)Tabel InputInput admin; t_bahan_baku (baca/tulis)Tabel Outputt_bahan_baku (tulis); notifikasi ke UI Dapur dan AdminAlgoritma1. Tampilkan form data bahan baku.
2. Admin mengisi nama, kategori, stok, satuan, stok minimum, tanggal masuk, tanggal kedaluwarsa.
3. Sistem menyimpan data ke t_bahan_baku.
4. Sistem mengevaluasi: jika stok < stok_minimum, set status_bahan = HampirHabis dan tampilkan notifikasi.
5. Sistem mengevaluasi: jika tanggal_kedaluwarsa <= 3 hari dari hari ini, tampilkan notifikasi Mendekati Kedaluwarsa.
6. Jika tanggal_kedaluwarsa telah terlewat, set status_bahan = Kedaluwarsa.
7. Jika kondisi normal, set status_bahan = Aman.
5.2.2 Spesifikasi Fungsi: Kelola Pesanan (FR-005, FR-006)
Identifikasi/NamaF-002: Kelola Pesanan DigitalDeskripsi IsiKasir memasukkan pesanan pelanggan ke dalam sistem; sistem memvalidasi kelengkapan dan meneruskan pesanan ke antrian dapur secara real-time.JenisForm Entry + Notifikasi Real-timeTabel InputInput kasir; t_pengguna (baca)Tabel Outputt_pesanan (tulis); notifikasi antrian ke layar dapur via WebSocketAlgoritma1. Kasir membuka halaman form pesanan.
2. Kasir mengisi: menu, topping, level pedas, jumlah, catatan (opsional).
3. Sistem memvalidasi kelengkapan data (menu, level pedas, jumlah wajib diisi).
4. Jika data tidak lengkap: tampilkan pesan validasi, hentikan proses.
5. Jika data lengkap: simpan ke t_pesanan dengan status = Menunggu.
6. Sistem mengirimkan notifikasi antrian ke layar dapur via WebSocket.
7. Kasir menerima konfirmasi pesanan berhasil dikirim.
5.2.3 Spesifikasi Fungsi: Kelola Produksi (FR-007, FR-008)
Identifikasi/NamaF-003: Kelola ProduksiDeskripsi IsiDapur memproses pesanan berdasarkan antrian FIFO; memperbarui status produksi; sistem mengurangi stok bahan baku secara otomatis setelah produksi selesai.JenisForm Entry + Proses Otomatis (Pengurangan Stok)Tabel InputInput dapur; t_pesanan (baca); t_bahan_baku (baca)Tabel Outputt_pesanan (update); t_produksi (tulis); t_bahan_baku (update stok)Algoritma1. Dapur membuka halaman antrian pesanan.
2. Dapur memilih pesanan berdasarkan urutan FIFO.
3. Dapur menekan tombol Proses; sistem: set status_pesanan = Diproses, buat record t_produksi dengan waktu_mulai = now().
4. Dapur memproses pesanan (persiapan bahan, pembakaran, packing).
5. Dapur menekan tombol Selesai; sistem: set status_pesanan = Selesai, set waktu_selesai = now().
6. Sistem secara otomatis mengurangi stok bahan baku terkait di t_bahan_baku berdasarkan referensi t_produksi_bahan.
7. Sistem mengevaluasi ulang status_bahan setelah pengurangan stok.
5.2.4 Spesifikasi Fungsi: Kelola Restock (FR-009)
Identifikasi/NamaF-004: Kelola RestockDeskripsi IsiDapur mencatat penambahan stok bahan baku; sistem memperbarui jumlah stok dan menyimpan riwayat restock.JenisForm EntryTabel InputInput dapur; t_bahan_baku (baca)Tabel Outputt_restock (tulis); t_bahan_baku (update stok)Algoritma1. Dapur membuka form restock bahan.
2. Dapur memilih nama bahan dari daftar (dropdown dari t_bahan_baku).
3. Dapur mengisi: jumlah tambah, tanggal restock, keterangan.
4. Sistem memvalidasi kelengkapan data (nama bahan dan jumlah tambah wajib diisi).
5. Jika valid: simpan ke t_restock, tambahkan jumlah_tambah ke stok bahan di t_bahan_baku.
6. Sistem mengevaluasi ulang status_bahan setelah penambahan stok.
5.2.5 Spesifikasi Fungsi: Cetak Laporan (FR-010 s.d. FR-012)
Identifikasi/NamaF-005: Cetak LaporanDeskripsi IsiSistem menghasilkan laporan operasional (stok bahan, produksi harian, restock) berdasarkan periode yang dipilih admin.JenisReport UITabel InputAdmin memilih jenis laporan dan periode; t_bahan_baku / t_pesanan / t_produksi / t_restock (baca)Tabel OutputData laporan ke dashboard adminAlgoritma1. Admin membuka halaman laporan.
2. Admin memilih jenis laporan: Stok Bahan / Produksi Harian / Restock.
3. Admin memilih periode waktu laporan.
4. Sistem mengambil data dari tabel terkait sesuai jenis dan periode.
5. Jika data tersedia: sistem menampilkan data laporan dalam format tabel.
6. Admin dapat mengunduh laporan dalam format yang tersedia.
7. Jika data tidak tersedia pada periode tersebut: sistem menampilkan pesan informasi.

6.4 Prosedur Umum Pengujian
6.4.1 Pengenalan dan Latihan
Sebelum pengujian formal dilakukan, seluruh anggota tim melakukan sesi pengenalan sistem selama 2 jam, mencakup:
• Pemahaman alur utama sistem (login, kelola bahan, input pesanan, produksi, restock, laporan).
• Pemahaman skenario pengujian yang telah disusun.
• Latihan penggunaan tools pengujian (Postman untuk API, browser devtools untuk UI).

6.4.2 Persiapan Awal
6.4.2.1 Persiapan Prosedural
• Memastikan server aplikasi berjalan dan dapat diakses dari semua perangkat uji.
• Memastikan database telah diisi dengan data master yang diperlukan (bahan baku, pengguna).
• Mencatat kondisi awal sistem (jumlah data di setiap tabel) sebagai baseline.

6.4.2.2 Persiapan Perangkat Keras
• Menghubungkan semua perangkat ke jaringan LAN yang sama.
• Setting IP statis untuk server agar semua klien dapat mengakses dengan URL konsisten.
• Memverifikasi koneksi layar dapur (tablet/monitor) ke jaringan LAN lokal.

6.4.2.3 Persiapan Perangkat Lunak
• Deploy aplikasi ke server dengan konfigurasi lingkungan pengujian (bukan production).
• Import data uji awal ke database menggunakan script SQL yang telah disiapkan.
• Buka Postman dan import koleksi endpoint API yang akan diuji.

6.4.3 Pelaksanaan
Strategi pelaksanaan pengujian dilakukan secara bertahap:
• Tahap 1 - Unit Testing: Pengujian fungsi individual menggunakan Jest untuk backend (validasi stok, evaluasi status bahan, logika pengurangan stok otomatis).
• Tahap 2 - Integration Testing: Pengujian alur antar modul menggunakan Postman (alur pesanan -> produksi -> kurangi stok; alur restock -> update stok).
• Tahap 3 - System Testing: Pengujian sistem secara end-to-end menggunakan browser (skenario lengkap dari login, input pesanan, proses produksi, hingga laporan).
• Tahap 4 - User Acceptance Testing (UAT): Pengujian oleh pengguna simulasi (kasir dan dapur non-teknis) untuk memvalidasi usability.

6.4.4 Pelaporan Hasil
Hasil pengujian akan didokumentasikan dalam tabel hasil uji dan diserahkan kepada dosen pengampu beserta kesimpulan kesesuaian dengan requirement yang telah ditetapkan.

• File README.md: Panduan instalasi dan pengembangan.
• Direktori /tests: Berisi script unit testing (Jest).
• Direktori /database: Berisi file schema.sql, seed.sql, dan skrip migrasi.

7.3 Syarat Pemaketan
Syarat pemaketan untuk distribusi perangkat lunak:
• Seluruh kode sumber dan aset dikemas dalam satu arsip ZIP/TAR.GZ.
• File .env.example disertakan sebagai template konfigurasi (tanpa nilai sensitif).
• Dokumentasi (DPPL dan BRD) disertakan dalam format PDF.
• Versi produk ditandai menggunakan Git Tag (format: v1.0.0).

7.4 Prosedur Konstruksi
Prosedur untuk membangun ulang perangkat lunak dari kode sumber:
• Prasyarat: Node.js 18+, MySQL 8.0+, npm 9+.

• Langkah 2: Jalankan 'npm install' untuk menginstal dependensi.

• Langkah 5: Jalankan 'npm run db:seed' untuk mengisi data master awal (menu bahan baku dan akun admin default).
• Langkah 6: Jalankan 'npm run build' untuk mengkompilasi kode frontend (jika menggunakan bundler).
